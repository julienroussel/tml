# Internationalization (i18n)

The Magic Lab uses next-intl for internationalization, supporting 7 locales.

## Supported Locales

| Code | Language | Status |
|---|---|---|
| `en` | English | Default, complete |
| `fr` | French | Planned |
| `es` | Spanish | Planned |
| `pt` | Portuguese | Planned |
| `it` | Italian | Planned |
| `de` | German | Planned |
| `nl` | Dutch | Planned |

## Locale Strategy

The app uses a dual strategy for locale detection:

### Marketing Pages -- URL-based Locale Routing

Public marketing pages use a `[locale]` URL segment for locale. Bare paths (`/faq`, `/privacy`) are 302-redirected by the proxy to locale-prefixed versions based on the user's `NEXT_LOCALE` cookie or `Accept-Language` header. The root path `/` redirects authenticated users to `/dashboard` and unauthenticated users to the locale-prefixed landing page.

```
/           --> 302 redirect to /dashboard (authenticated) or /en (unauthenticated)
/en         --> English homepage (static)
/fr         --> French homepage (static)
/es/faq     --> Spanish FAQ page (static)
```

All 7 locales × 3 pages = 21 statically generated pages at build time (`● SSG` in build output). This approach is SEO-friendly: each locale has its own URL, enabling proper `hreflang` tags and search engine indexing.

### Auth Pages -- Cookie-Based (Dynamic)

Auth pages (sign-in, sign-up) use the `NEXT_LOCALE` cookie for locale detection. They are server-rendered dynamically (not statically generated) so the server can read the cookie and render the Neon Auth form in the user's chosen language. A locale toggle and theme toggle are available in the page header.

The Neon Auth UI is localized via the `auth` namespace in the message files. The `NeonAuthLocalizedProvider` component extracts these keys and passes them to `NeonAuthUIProvider` as its `localization` prop.

### App Pages -- Cookie-Based

```
/dashboard  --> Locale from user_preferences.locale or cookie
```

This avoids cluttering app URLs with locale prefixes while still respecting the user's language choice. The preference is stored in the `user_preferences` table and synced via cookie.

## Message Catalog Structure

Messages are organized by feature module and shared components:

```
messages/
  en/
    common.json         # Shared strings (buttons, labels, errors)
    navigation.json     # Sidebar, header, breadcrumbs
    dashboard.json      # Dashboard module
    improve.json        # Improve module
    train.json          # Train module
    plan.json           # Plan module
    perform.json        # Perform module
    enhance.json        # Enhance module
    collect.json        # Collect module
    settings.json       # Settings
    marketing.json      # Landing page, about, FAQ
    auth.json           # Neon Auth UI strings (sign-in, sign-up, OTP, social login)
    email.json          # Transactional email and unsubscribe page strings
  fr/
    common.json
    ...
```

> **Note**: The `auth` namespace keys use UPPER_SNAKE_CASE (e.g., `SIGN_IN`, `EMAIL_OTP_SEND_ACTION`) to match the `AuthLocalization` type from `@neondatabase/auth`. These are extracted at runtime by `src/i18n/auth-localization.ts` and passed to `NeonAuthUIProvider`.

### Email Namespace

The `email` namespace localizes transactional emails (welcome email) and the unsubscribe page. Unlike other namespaces, email translations are loaded **server-side outside of next-intl's request context** (email rendering has no access to `useTranslations` or `getTranslations`).

- **Helper**: `src/i18n/email-translations.ts` exports `getEmailTranslations(locale)` which dynamically imports the locale JSON and extracts the `email` namespace.
- **Interpolation**: Uses simple `{name}` placeholder replacement via `interpolate()` — not ICU MessageFormat.
- **Fallback**: If a locale's email namespace is incomplete or missing, falls back to the default locale (`en`).
- **Keys** (21 total): `welcomeSubject`, `welcomePreview`, `welcomeGreeting`, `welcomeGreetingAnonymous`, `welcomeBody`, `welcomeFeature*`, `welcomeCta`, `welcomeFooter`, `unsubscribeTitle`, `unsubscribeDescription`, `unsubscribeButton`, `unsubscribeSuccess*`, `unsubscribeError*`, `errorTitle`.
- **Locale seeding**: New users get their locale seeded from the `NEXT_LOCALE` cookie at registration time, so the welcome email is sent in the language they selected on the auth page.

### Message Format

Messages use ICU MessageFormat syntax via next-intl:

```json
{
  "greeting": "Hello, {name}!",
  "sessions": "You have {count, plural, =0 {no sessions} one {# session} other {# sessions}}.",
  "lastPractice": "Last practice: {date, date, medium}"
}
```

## Translation Workflow

1. **Development**: Write all strings in English message catalogs
2. **Extraction**: New keys are added to the English catalog first
3. **Translation**: Translate to other locales (manual or automated)
4. **Review**: Native speakers review translations
5. **Integration**: Translated catalogs are committed to the repo

## SEO (hreflang)

Marketing pages include `hreflang` link tags for search engines:

```html
<link rel="alternate" hreflang="en" href="https://themagiclab.app/en" />
<link rel="alternate" hreflang="fr" href="https://themagiclab.app/fr" />
<link rel="alternate" hreflang="es" href="https://themagiclab.app/es" />
<link rel="alternate" hreflang="pt" href="https://themagiclab.app/pt" />
<link rel="alternate" hreflang="it" href="https://themagiclab.app/it" />
<link rel="alternate" hreflang="de" href="https://themagiclab.app/de" />
<link rel="alternate" hreflang="nl" href="https://themagiclab.app/nl" />
```

Each marketing page exports a `generateMetadata` function that includes `alternates.languages` for all 7 locales. Next.js converts these into `<link rel="alternate" hreflang="...">` tags automatically.

## Offline Availability

For offline-first support, message catalogs are:

- **Bundled at build time** into the client bundle via next-intl's compilation step
- **Cached by the service worker** as part of the app shell
- Available immediately without network requests

This ensures the UI remains fully localized even when the device is offline.

## Usage in Components

### Server Components

```typescript
import { getTranslations } from "next-intl/server";

export default async function TrainPage() {
  const t = await getTranslations("train");
  return <h1>{t("title")}</h1>;
}
```

### Client Components

```typescript
"use client";
import { useTranslations } from "next-intl";

export function GoalCard() {
  const t = useTranslations("train");
  return <p>{t("goalDescription")}</p>;
}
```
