# Sync Engine Reference

**CRITICAL — Offline-first is a load-bearing requirement.** App must boot and operate with no network after first load. PowerSync's local SQLite is the primary read source. The service worker (`public/sw.js`) must cache PowerSync WASM + workers under `/powersync/`; `shouldBypass()` skips only live API traffic (sync/auth/analytics), never static assets.

| Layer | Path | Status |
|---|---|---|
| Server schema | `src/db/schema/` (Drizzle, Neon) | **source of truth** |
| Client schema | `src/sync/schema.ts` | generated |
| Sync config | `powersync/sync-config.yaml` (Streams edition 3) | generated |
| Column allowlist | `src/sync/synced-columns.ts` | generated |

- **Write path**: Component → `execute()` → upload queue → `/api/powersync/upload` → Drizzle → Neon.
- **Read path**: Neon → PowerSync Cloud → client SQLite → `useQuery()` → React.
- **Conflict resolution**: last-write-wins on `updated_at`.
- **Deletion**: soft-delete with `deleted_at`, 30-day hard-delete cleanup.
- **Mutation scoping**: connector forces authenticated `user_id` server-side — never trust from client.
- **Error semantics**: 4xx = permanent (mutation dropped); 5xx/network = retry.

## Sync Config Codegen

Generated by `scripts/generate-sync.ts` from the Drizzle schema. **Never hand-edit** the generated artifacts.

- Run `pnpm sync:generate` after any change under `src/db/schema/**`.
- `pnpm sync:check` runs in pre-commit (Lefthook) and CI.
- Server-only tables (`users`, `user_preferences`, `push_subscriptions`) are in `SERVER_ONLY_TABLES` in `scripts/generate-sync.ts`. Tables sync by default — add to denylist only if a table must never reach the client.
- Generator fails fast on unknown Drizzle `columnType`. Add a mapping in `mapColumnType` when that happens.

## PowerSync Instance Topology

There are **two PowerSync instances under one PowerSync project** (`POWERSYNC_PROJECT_ID`):

| Role | Instance ID prefix | URL | Used by |
|---|---|---|---|
| **Dev** | `69b6f273…` | `…powersync.journeyapps.com` set in development Vercel env's `NEXT_PUBLIC_POWERSYNC_URL` | local dev (`.env.local`), Vercel "development", Vercel "preview" |
| **Prod** | `69b6f2747c4f8b306a1bc349` | production Vercel env's `NEXT_PUBLIC_POWERSYNC_URL` | production `themagiclab.app` |

They are **independent runtimes** — separate sync-rules state, separate bucket store, separate JWT trust configs. Rules deployed to one do not propagate to the other.

`deploy-sync.yml` routes by event:

- **PR to main** (paths match) → deploys to the **dev** instance. All dev deploys share a single concurrency group so two PRs touching sync paths cannot race; a fresh PR push cancels the in-flight dev run (possibly from another PR). The canceller deploys ITS OWN sync config — the cancelled PR's config is NOT auto-re-deployed; its author has to push a new commit to retest on dev.
- **Push to main** (post-merge) → deploys to the **prod** instance. Concurrency group ensures prod deploys serialize.

Required GitHub repository variables: `POWERSYNC_INSTANCE_ID_DEV`, `POWERSYNC_INSTANCE_ID_PROD`, `POWERSYNC_PROJECT_ID`. Required secret: `POWERSYNC_ADMIN_TOKEN` (one PAT covers all instances in the project). Fork PRs and Dependabot PRs are skipped (no access to the admin token).

### GitHub Environment gating

The `deploy` job is bound to a GitHub Environment (`environment: production` on push/dispatch, `environment: preview` on PR). **Both environments must exist in the repository's Settings → Environments before the workflow can run** — if they don't, the deploy job fails at startup with `Could not find the environment …`, breaking the next push-to-main prod deploy AND the next PR's dev deploy. Create them before merging this change.

The GitHub Environment is named `preview` (matching Vercel's terminology for PR deployments) even though it deploys to the **dev** PowerSync instance — the two names describe different layers (the GitHub deploy target vs. the PowerSync runtime). A PR-event deploy targets `vars.POWERSYNC_INSTANCE_ID_DEV` regardless of the GitHub Environment name.

Today `POWERSYNC_ADMIN_TOKEN` is a **repository secret** (not environment-scoped), so the environment binding currently provides an **approval gate, not secret isolation**:

- A required reviewer must click "Approve and deploy" before the job runs and the PAT enters the job's env.
- The deploy is audit-logged with the approver's identity.
- A malicious PR that edits `scripts/powersync.ts` to exfiltrate the PAT will still succeed *if* a reviewer approves the run without reading the diff carefully.

Recommended environment configuration today:

- **`production`** — Required reviewers: at least one maintainer. Deployment branches: `main` only. Closes the workflow_dispatch-from-feature-branch hole; gates prod deploys on human approval.
- **`preview`** — Required reviewers: at least one maintainer (recommended). No branch restriction. Required-reviewer is what blocks a same-repo contributor from silently running secret-bearing code via a PR push; without it, the gate degrades to the `if:` filter in the deploy job (fork + Dependabot still blocked, same-repo contributors still trusted on push).
  - **Trade-off note**: Required reviewers on `preview` means every same-repo PR push that touches a sync-related path pauses for manual approval. If your team values fast PR iteration over the approval gate on dev deploys, leave `preview` without required reviewers and rely on the deploy job's `if:` filter (which still blocks fork PRs and Dependabot from secret-bearing runs). Production should always have required reviewers regardless.

To upgrade to **true secret isolation** (the stronger posture):

1. Provision a second PAT in the PowerSync dashboard scoped to the dev instance only.
2. Move both PATs from repository secrets to environment secrets: `POWERSYNC_ADMIN_TOKEN` in the `production` environment (prod-scoped PAT), `POWERSYNC_ADMIN_TOKEN` in the `preview` environment (dev-scoped PAT). Remove the repository-level secret afterwards.
3. After this change, a malicious PR can at worst burn the dev PAT (which can only deploy to the dev instance); the prod PAT is never reachable from any PR-event run, regardless of reviewer approval.

### Symptoms of a stale / un-deployed instance

If the deployed sync config is missing or drifted from `powersync/sync-config.yaml`, clients see all of these at once:

- `SyncStatus` pill reports `degraded` (orange) — not `online`. Orange is distinct from the amber `pendingChanges` pill: amber = "writes queued, will resolve itself"; orange = "server returned no buckets, data may be wiped on next sync". Powered by `useBucketHealth()` in `src/sync/use-bucket-health.ts`, which counts non-`$local` rows in `ps_buckets`.
- `/dev/sync-status` (dev-only diagnostic route) shows `ps_buckets` containing only the `$local` placeholder, `ps_oplog` count = 0, but `ps_stream_subscriptions` lists the streams as active.
- Local writes appear in feature pages momentarily then disappear on the next sync cycle (server confirms the empty bucket → local writes get wiped).
- `/activity` and dashboard count cards show empty even for users who have data in Neon.
- `gh run list --workflow=deploy-sync.yml --limit 5` shows recent failed runs (or no runs since the schema change).

Resolution recipe: confirm CI deploy succeeded for the affected instance label (`dev` or `prod`) in `gh run view <run-id>`. If the failure is `Deploy the instance, with powersync deploy, before validating sync config.`, the workflow's instance-deploy step is the fix — see `.github/workflows/deploy-sync.yml` for the order (instance deploy → validate → sync-config deploy). For an out-of-band manual recovery: `POWERSYNC_INSTANCE_ID=… POWERSYNC_PROJECT_ID=… POWERSYNC_ADMIN_TOKEN=… pnpm exec tsx scripts/powersync.ts deploy && pnpm sync:deploy`.

### Partial-state recovery

Deploy order is `instance deploy` → `sync:validate` → `sync:deploy`. If `sync:validate` fails between the first and third steps, the instance has been re-deployed (new connector / auth config) but sync rules remain on the previous version. The instance continues to serve old rules to clients — no immediate data corruption — until the workflow is re-run. Fix the config error and re-trigger (push or manual dispatch); the instance-deploy step is idempotent.

### Branch protection contract

Mark the `validate` job in `deploy-sync.yml` as a **required** status check — it runs for all PRs including fork PRs and Dependabot. Do **NOT** mark the `deploy` job required. The `deploy` job's `if:` produces GitHub "skipped" status for fork/Dependabot PRs (no access to `POWERSYNC_ADMIN_TOKEN`), and GitHub treats "skipped" as neither pass nor fail — required-status-check rules would permanently block merge.

**Prerequisite before flipping `validate` to required**: the workflow currently uses workflow-level `concurrency.cancel-in-progress: true` on PR events, which cancels an in-flight `validate` when a fresh PR push arrives. GitHub treats cancelled checks the same as "skipped" for required-check rules, so marking `validate` required without first splitting concurrency would permanently block merge on rapid back-to-back pushes. Before flipping the required flag, move the `concurrency:` block from workflow level onto the `deploy` job only (validate runs with no concurrency block; deploy keeps `cancel-in-progress` on PR events). The workflow comment above the `concurrency:` key documents the same trade-off.

## Sync Config Deployment

Deployed via the PowerSync CLI from GitHub Actions.

- **Workflow**: `.github/workflows/deploy-sync.yml` runs on PR + push to `main` (paths-based). The target instance is decided at runtime from `github.event_name`: PR → dev, push → prod. Sequence per run: `pnpm exec tsx scripts/powersync.ts deploy` (instance) → `pnpm sync:validate` → `pnpm sync:deploy` (sync-config). The instance-deploy step is non-optional — without it, sync-config validation fails with `Deploy the instance, with powersync deploy, before validating sync config.`
- **Manual deploy**: `POWERSYNC_ADMIN_TOKEN=... POWERSYNC_INSTANCE_ID=... POWERSYNC_PROJECT_ID=... pnpm sync:deploy`. **Do not paste the PAT inline** — it lands in `~/.zsh_history` / `~/.bash_history` unencrypted. Prefer `direnv` (`.envrc` excluded from git), `op run --env-file=.env.sync -- pnpm sync:deploy`, or prefix with a leading space when `HISTCONTROL=ignorespace` is set.
- **Credentials** (all `POWERSYNC_*`-prefixed — thin `scripts/powersync.ts` wrapper maps to CLI's bare names):
  - `POWERSYNC_ADMIN_TOKEN` — GitHub **secret**, PAT from https://dashboard.powersync.com/account/access-tokens.
  - `POWERSYNC_INSTANCE_ID` / `POWERSYNC_PROJECT_ID` — GitHub repository **variables** (not secrets; visible in logs). Discover via `powersync fetch instances --output=json`.
  - `POWERSYNC_ORG_ID` — only if the PAT spans multiple orgs.
- **Rollback**: revert the commit and re-run. No native CLI rollback.
- **Dashboard is view-only** — never edit sync rules there. Changes must flow through PR.

## Preview Deployment Sync Coverage

Vercel preview deployments (`tml-*-jaro-labs.vercel.app` — `e2e.yml` selects them by the `jaro-labs` team slug in the deployment URL) **cannot** validate server-side sync. Preview Vercel environments point `NEON_AUTH_BASE_URL` at a different Neon Auth instance than production; the production PowerSync Cloud instance trusts only production's JWKS + audience, so preview-issued JWTs are rejected at the WebSocket handshake (`PSYNC_S2105`). The sync stream never connects — and because PowerSync only calls `uploadData()` while the sync stream is connected, the upload queue never drains either.

- **Preview E2E (`e2e.yml`) validates**: local-first SQLite reads/writes, offline behavior, auth, and that the PowerSync client boots. (PWA / service-worker specs are localhost-only — `playwright.config.ts` excludes the `pwa` project whenever `BASE_URL` is set.)
- **It does NOT validate**: server→client sync or the client→server write round-trip — those are verified only against production (`themagiclab.app`).
- **E2E gate**: use `waitForSyncReady` (WASM/workers up — survives the rejected sync stream). `waitForSynced` (full server sync) hangs on previews; see `e2e/helpers.ts`.

Resolved as a documented limitation in #302 (Option D). Real preview sync coverage — a dedicated preview PowerSync instance — is tracked in #312.
